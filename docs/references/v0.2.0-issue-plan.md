## 用語と凡例

- **Epic**: Phase 単位の親 Issue。複数の子 Issue を束ねる。close 前に **Phase E2E gate** を必ず通す
- **子 Issue**: 1 PR で close 可能な粒度。`### Issue X.Y` 見出しで列挙
- **Phase E2E gate**: 親 close 直前に置く統合確認 Issue (`Issue P{X}-E2E`)。子 Issue 全 merge 前提に E2E fixture と SPEC gate を緑化
- **Issue ID**: `<Phase 番号>.<連番>` (例: `1.1` / `2A.3` / `E.2`)
- **依存**: `blocked-by: <ID>` で他 Issue 完了前提を表現。並列可は明記
- **見積**: S (1 日以内) / M (2〜3 日) / L (1 週間以上) の主観値
- **failure.code 標識**: 新規 = `cross-cutting.md` §1 由来 (`effort_unsupported` / `preset_path_traversal` / `preset_blacklist_hit`) / 既存再利用 = SPEC §4.2.1.1 既出
- **SPEC 同 PR 更新責務**: 各 Issue の最後に SPEC.md §X.Y への追記責務を明記 (`cross-cutting.md` §4 / 既存 PLAN 重要原則 10 準拠)

子 Issue なし。全子 Issue 共通の **横断ルール**を以下に定義し、各 Issue は本ルールに従う責務を負う。

## P0.1 SPEC trace 同 PR 更新義務 (cross-cutting.md §4)

新 `failure.code` を導入する PR は **必ず同 PR で**:

1. `packages/core/src/failure-codes.ts` の `failureCodes` 配列に追加
2. `docs/SPEC.md` §4.2.1.1 表に行追加 (state / phase / 意味)
3. `docs/SPEC.md` §10.2.2.2 失敗系 audit kind リストに `- \`<code>\`` 行追加
4. `bash scripts/check-trace.sh` ローカル実行で 1:1 trace 緑確認
5. `packages/core/src/logger.ts` の `failureAuditKinds = failureCodes` 規約 (logger.ts:24) で型推論自動同期を確認

新 操作系 audit kind を導入する PR は §10.2.2.1 表のみ追加し、`packages/core/src/logger.ts` の `operationalAuditKinds` 配列 (logger.ts:26-41) に追加 (1:1 trace 対象外)。

## P0.2 既存 E34 condition 改訂責務 (cross-cutting.md §3.1)

Issue 1.7 で SPEC §5.1 表 E34 (`prompt_contract_violation`) の condition 列を以下のように **必ず同 PR で改訂**:

- 旧: 「prompt_contract 違反 (`status=need_input` で `default` 欠落 等)」即時 `failed`
- 新: 「prompt_contract 違反 + `runtime.phase_self_correct_done=true`」で `failed`。`false` 状態の 1 回目違反は **state-machine 経由せず** workflows 内 retry orchestrator (`runWithSelfCorrection`) で field 更新 + audit kind `phase_self_correct` 発火

## P0.3 assets-hygiene 緑化義務 (cross-cutting.md §8)
