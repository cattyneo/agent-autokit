## 用語と凡例

- **Epic**: Phase 単位の親 Issue。複数の子 Issue を束ねる。close 前に **Phase E2E gate** を必ず通す
- **子 Issue**: 1 PR で close 可能な粒度。`### Issue X.Y` 見出しで列挙
- **Phase E2E gate**: 親 close 直前に置く統合確認 Issue (`Issue P{X}-E2E`)。子 Issue 全 merge 前提に E2E fixture と SPEC gate を緑化
- **Issue ID**: `<Phase 番号>.<連番>` (例: `1.1` / `2A.3` / `E.2`)
- **依存**: `blocked-by: <ID>` で他 Issue 完了前提を表現。並列可は明記
- **見積**: S (1 日以内) / M (2〜3 日) / L (1 週間以上) の主観値
- **failure.code 標識**: 新規 = `cross-cutting.md` §1 由来 (`effort_unsupported` / `preset_path_traversal` / `preset_blacklist_hit`) / 既存再利用 = SPEC §4.2.1.1 既出
- **SPEC 同 PR 更新責務**: 各 Issue の最後に SPEC.md §X.Y への追記責務を明記 (`cross-cutting.md` §4 / 既存 PLAN 重要原則 10 準拠)

# Epic P0: Cross-cutting / SPEC trace 同 PR 更新規約

子 Issue なし。全子 Issue 共通の **横断ルール**を以下に定義し、各 Issue は本ルールに従う責務を負う。

## P0.1 SPEC trace 同 PR 更新義務 (cross-cutting.md §4)

新 `failure.code` を導入する PR は **必ず同 PR で**:

1. `packages/core/src/failure-codes.ts` の `failureCodes` 配列に追加
2. `docs/SPEC.md` §4.2.1.1 表に行追加 (state / phase / 意味)
3. `docs/SPEC.md` §10.2.2.2 失敗系 audit kind リストに `- \`<code>\`` 行追加
4. `bash scripts/check-trace.sh` ローカル実行で 1:1 trace 緑確認
5. `packages/core/src/logger.ts` の `failureAuditKinds = failureCodes` 規約 (logger.ts:24) で型推論自動同期を確認

新 操作系 audit kind を導入する PR は §10.2.2.1 表のみ追加し、`packages/core/src/logger.ts` の `operationalAuditKinds` 配列 (logger.ts:26-41) に追加 (1:1 trace 対象外)。

## P0.2 既存 E34 condition 改訂責務 (cross-cutting.md §3.1)

Issue 1.7 で SPEC §5.1 表 E34 (`prompt_contract_violation`) の condition 列を以下のように **必ず同 PR で改訂**:

- 旧: 「prompt_contract 違反 (`status=need_input` で `default` 欠落 等)」即時 `failed`
- 新: 「prompt_contract 違反 + `runtime.phase_self_correct_done=true`」で `failed`。`false` 状態の 1 回目違反は **state-machine 経由せず** workflows 内 retry orchestrator (`runWithSelfCorrection`) で field 更新 + audit kind `phase_self_correct` 発火

state-machine.ts の `TransitionEvent` union への新規追加は **なし** (cross-cutting.md §3.2)。

## P0.3 assets-hygiene 緑化義務 (cross-cutting.md §8)

`packages/serve/` (Issue 2A.2 で新設) や `packages/dashboard/` (将来 Phase 2B) を追加する PR は `bash scripts/check-assets-hygiene.sh` を **緑化**:

- `packages/cli/package.json` `"private": true` 維持
- `packages/cli/dist/bin.js` に `workspace:` 指定子 / unresolved `@cattyneo/autokit-{core,workflows,...}` import が残らない (= `bun build` で self-contained bundle)
- `bun pm pack --dry-run` 出力に `__MACOSX` / `.DS_Store` / `.claude/state` / `.claude/sessions` / `.claude/credentials` / `.codex/auth` / `.codex/credentials` / `.env*` / `*.pem` / `id_rsa*` が含まれない

## P0.4 後方互換義務

責務を 2 段に分ける:

1. **既存 e2e 全緑維持責務** — Issue 1.2 / Issue 1.7 / Issue 3.1 共通。新機能 / 改訂を入れた後も既存 fixture / e2e (`e2e/runners/full-run.test.ts` 等) が緑であること。
2. **`zod default` + `z.preprocess` 使い分け責務 (yaml schema migration)** — **Issue 1.2 のみ**。zod `default()` は値補完、key rename / structure migration は `z.preprocess` で実装 (Phase 1 §4.3.1)。Issue 3.1 (assets-writer) は file-system copy / rollback の共通化であり yaml schema migration を担当しない。

## P0.5 ラベル規約 (Issue 起票時)

GitHub Issue 化時は以下ラベルを付与:

- 親 Epic: `type:epic` / `phase:p1` 〜 `phase:p4` / `phase:p2b` / `phase:docs` / `phase:e2e`
- 子: `type:feature` または `type:test` / `phase:pX` / `area:core` / `area:cli` / `area:runner` / `area:serve` / `area:preset` / `area:assets`
- 重要原則違反検出系: `gate:trace` / `gate:hygiene`
- 既存 v0.1.0 ラベル (`mvp:v0.1` / `sprint:S0〜S7`) は本計画では使用しない

## Phase E2E gate harness policy

P1-E2E / P2A-E2E / P3-E2E / P4-E2E / E.* の required gate は live Claude/Codex provider subprocess を呼ばない。runner 境界は fake `WorkflowRunner` / runner subprocess mock で注入し、GitHub CLI 境界は `fakeGh` で固定する。Claude/Codex 実行を伴う provider-backed test は `--allow-model-calls` 等の明示 opt-in + optional matrix のみに分離する。required gate が `ANTHROPIC_API_KEY` / `OPENAI_API_KEY` / `CODEX_API_KEY` set 状態で live runner dispatch へ進む場合は fail-closed とする。
